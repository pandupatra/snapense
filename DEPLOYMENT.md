# Deployment Guide

This document explains how to deploy Snapense to a VPS using Docker, Nginx, and GitHub Actions.

## Overview

The deployment stack:
- **Docker & Docker Compose** for containerization and process management
- **Nginx** as reverse proxy with SSL termination
- **Certbot** for automatic Let's Encrypt SSL certificates
- **GitHub Actions** for push-to-deploy automation
- **SQLite** database persisted on host via bind mount

## Prerequisites

- A VPS with Ubuntu 22.04+ and root access
- A domain name pointing to your VPS IP
- GitHub repository with your code

## Initial VPS Setup

Run the deployment script on your VPS:

```bash
# Clone your repository
git clone <your-repo-url> /var/www/snapense
cd /var/www/snapense

# Run the deployment script
sudo bash deploy.sh <your-domain> <your-email>
```

Example:
```bash
sudo bash deploy.sh snapense.pandupatra.site admin@example.com
```

This script will:
1. Install Docker, Docker Compose, Nginx, and Certbot
2. Configure the production environment file
3. Build and start the application container
4. Configure Nginx as a reverse proxy
5. Obtain and configure SSL certificates
6. Set up the firewall (UFW)
7. Configure automatic database backups

## Environment Variables

After running `deploy.sh`, update `.env.production` with your actual API keys:

```bash
sudo nano /var/www/snapense/.env.production
```

Required variables:
- `BETTER_AUTH_SECRET` - auto-generated by deploy.sh
- `GOOGLE_CLIENT_ID` - from Google Cloud Console
- `GOOGLE_CLIENT_SECRET` - from Google Cloud Console
- `GEMINI_API_KEY` - from Google AI Studio

Then restart the container:
```bash
cd /var/www/snapense
sudo docker compose restart
```

## Manual Updates

To manually update the application:

```bash
ssh user@your-vps-ip
cd /var/www/snapense
bash update.sh
```

The `update.sh` script will:
1. Pull the latest code from git
2. Rebuild and restart the Docker container
3. Clean up unused Docker images

## GitHub Actions Setup

### Step 1: Configure Secrets

Go to your GitHub repository → Settings → Secrets and variables → Actions → New repository secret

Add the following secrets:

| Secret Name | Description | Example |
|-------------|-------------|---------|
| `VPS_HOST` | Your VPS IP address or domain | `123.45.67.89` or `snapense.pandupatra.site` |
| `VPS_USER` | SSH username | `root` or `ubuntu` |
| `VPS_PASSWORD` | SSH password | `your-secure-password` |
| `VPS_PORT` | SSH port (optional, defaults to 22) | `22` |

### Step 2: Deploy

Push to the `main` branch:

```bash
git add .
git commit -m "Deploy new feature"
git push origin main
```

GitHub Actions will automatically:
1. SSH into your VPS
2. Pull the latest code
3. Rebuild and restart the Docker container

## Docker Commands

```bash
# Check container status
docker compose ps

# View logs
docker compose logs -f

# Restart application
docker compose restart

# Rebuild and restart
docker compose up --build -d

# Stop application
docker compose down
```

## Database Backups

Backups are automatically created daily at 2:00 AM and stored in `/var/backups/snapense/`.

### Manual Backup

```bash
/usr/local/bin/snapense-backup.sh
```

### Restore from Backup

```bash
# Stop the container
cd /var/www/snapense
docker compose down

# Restore from a backup file
sudo cp /var/backups/snapense/db_YYYYMMDD_HHMMSS.sqlite /var/www/snapense/.data/db.sqlite

# Start the container
docker compose up -d
```

Backups older than 7 days are automatically deleted.

## SSL Certificates

SSL certificates are automatically managed by Certbot. They renew automatically via a systemd timer.

To manually renew:
```bash
sudo certbot renew --nginx
```

## Troubleshooting

### Container won't start

```bash
cd /var/www/snapense
docker compose logs
```

### Permission denied on .data directory

```bash
sudo chown -R $(whoami):$(whoami) /var/www/snapense/.data
```

### Nginx errors

```bash
sudo nginx -t
sudo systemctl status nginx
```

### Port already in use

If port 3000 is already in use:
```bash
sudo lsof -i :3000
sudo kill -9 <PID>
```

## Security Best Practices

1. **Use a dedicated SSH user** for GitHub Actions (not root if possible)
2. **Restrict SSH key permissions** - limit to only run `update.sh`
3. **Keep `.env.production` secure** - never commit it to git
4. **Rotate secrets regularly** - update API keys periodically
5. **Monitor deployment logs** - check GitHub Actions for suspicious activity
6. **Use branch protection** - require PR reviews before merging to main

## Rollback

If a deployment breaks, the old container continues running until the new one builds successfully. To manually rollback:

```bash
ssh user@your-vps-ip
cd /var/www/snapense
git log --oneline
git reset --hard <previous-commit-hash>
bash update.sh
```
